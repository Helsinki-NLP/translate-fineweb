| The other day, I found myself explaining different types of encryption, how they work, and what they are used for to someone in my lab implementing a secure system. During this process, I ended up creating a series of fancy diagrams in draw.io - so I thought I'd write it up into a proper demystification blog post.<br/><br/>To start us off here, let's define encryption. Encryption is the process of transforming a given input block of data (of an arbitrary data) using some kind of secret key into a form that is then completely unreadable. Any adversary obtaining a block of encrypted data encrypted with a suitably strong key (and algorithm) is not able to read or understand the data at all - except perhaps infer its original length.<br/><br/>Conversely, decryption is the process of undoing the encryption process with the same (or different, in some cases) key to get back the original data.<br/><br/>For purpose of this blog post, we will assume:<br/><br/>The encryption algorithms in question are perfect with no known weaknesses<br/><br/>Keys used to encrypt and/or decrypt are very strong and can't be cracked<br/><br/>Each of these are fields in their own right that could quite easily take many blog posts to fully explore.<br/><br/>From the perspective of a developer, there are 3 different basic places one needs to aware of. Others certainly exist, but to avoid making this post too long I'll just be covering the following 3:<br/><br/>Device encryption<br/><br/>Transport layer encryption<br/><br/>End-to end encryption<br/><br/>If there's any other encryption scheme you'd like me to cover, please leave a comment below and I'll try my best to explain it in a separate post.<br/><br/>Device encryption<br/><br/>First up is device encryption. Most modern operating systems for phones and PCs alike support device encryption:<br/><br/>Windows<br/><br/>Linux<br/><br/>Android<br/><br/>iOS<br/><br/>Not sure on macOS since I don't own one, but I'd be surprised if it didn't. The purpose of device encryption is that when the device is powered off, all data is stored physically on disk in an encrypted format, making it unreadable should the device be physically stolen - thereby protecting all data stored on it.<br/><br/>This is accomplished in a layered fashion. Let's explain it with a diagram:<br/><br/>Although they may have different names for it, most operating systems back a concept of a "block device". Such a device is capable of storing a given number of bytes of data. Such devices need not be physical disks: they can instead be virtual. For example, zram presents block devices that store data compressed in RAM.<br/><br/>We can make use of this to encrypt hard drives. An encryption layer such as LUKS on Linux presents a virtual block device to the operating system which encrypts all data written to it before saving them back to some physical disk by which it is backed.<br/><br/>On boot, the encryption layer is initialised by the operating system and it asks the user for a password. Upon being given the correct password, the encryption layer is activated, and the operating system can then both request data blocks from the virtual block device (which causes the encryption layer to fetch the encrypted block from disk and then decrypt it before passing it to the requester) and write data blocks back to the virtual block device (whereby the encryption layer will encrypt the new data block before writing it to disk).<br/><br/>Even operating systems such as Windows (e.g. Bitlocker) and iOS which don't expose block devices in the same way as Linux does, the same principles I've explained here apply.<br/><br/>When the device is powered off, the key that was being stored in memory is wiped (it's stored in RAM, and RAM requires power to store data) and the data is secured.<br/><br/>Transport layer encryption<br/><br/>Another place encryption is commonly encountered in when transferring data to and from remote hosts over the Internet. După ce li se dă parola corectă, stratul de criptare este activat, iar sistemul de operare poate solicita blocuri de date de la dispozitivul de bloc virtual (ceea ce determină stratul de criptare să recupereze blocul criptat de pe disc și apoi să îl decripteze înainte de a-l trece solicitantului) și să scrie blocuri de date înapoi pe dispozitivul de bloc virtual (prin care stratul de criptare va cripta noul bloc de date înainte de a scrie pe disc).<br/><br/>Chiar și sistemele de operare precum Windows (de exemplu Bitlocker) și iOS care nu expun dispozitivele blocate în același mod ca Linux, se aplică aceleași principii pe care le-am explicat aici.<br/><br/>Când dispozitivul este oprit, cheia care a fost stocată în memorie este ștearsă (este stocată în RAM, iar RAM necesită energie pentru a stoca date) și datele sunt securizate.<br/><br/>Criptarea stratului de transport<br/><br/>Un alt loc de criptare este frecvent întâlnit în atunci când transferul de date la și de la gazde la distanță pe Internet.
