The datomic.Connection.txReportQueue method provides a<br/>method to get a queue of transaction notifications.<br/><br/>The queue delivers a report for every transaction submitted while a<br/>peer is connected to the database, even those submitted by other<br/>peers. The reports are instances of the datomic.TxReport type, which<br/>provides access to the value of the database before and after the<br/>transaction was applied, and the set of facts added by the<br/>transaction.<br/><br/>Once your code gets the transaction report queue, reports will be<br/>added to it. It is the responsibility of your peer code to empty the<br/>queue. When you are done monitoring the queue, you must remove it by<br/>calling the datomic.Connection/removeTxReportQueue method so it<br/>doesn't continue to consume memory.<br/><br/>This example shows how to connect to the notification queue, and<br/>retrieve a transaction report from it.<br/><br/>queue = conn.txReportQueue();<br/>report = (Map) queue.poll();<br/><br/>The report map contains the following keys, which are static members<br/>of the datomic.Connection class:<br/><br/>key<br/><br/>usage<br/><br/>DB_BEFORE<br/><br/>database value before the transaction<br/><br/>DB_AFTER<br/><br/>database value after the transaction<br/><br/>TX_DATA<br/><br/>datoms produced by the transaction<br/><br/>TEMPIDS<br/><br/>used to resolve temporary ids<br/><br/>The TX_DATA member of a transaction report contains the set of<br/>datoms created by a transaction (both assertions and retractions). By default, this entity has one attribute,<br/>:db/txInstant, whose value is the instant that the transaction was<br/>processed. In addition, every datom in a transaction refers to the<br/>transaction entity through the :tx field of datom.<br/><br/>You can add additional attributes to a transaction entity to capture<br/>other useful information, such as the purpose of the transaction, the<br/>application that executed it, the provenance of the data it added, or<br/>the user who caused it to execute, or any other information that might<br/>be useful for auditing purposes.<br/><br/>To annotate the current transaction, simply include an add statement<br/>(either map or list) that uses a temporary id from the :db.part/tx<br/>partition. This capability enables initial imports of<br/>existing data that has its own timestamps.<br/><br/>Redundancy Elimination<br/><br/>A datom is redundant with the current value of the database if there<br/>is a matching datom that differs only by transaction id. If a<br/>transaction would produce redundant datoms, those datoms are filtered<br/>out, and do not appear a second time in either the indexes or the<br/>transaction log.<br/><br/>Transactions are never entirely redundant, because the transaction's<br/>:db/txInstant is always a new fact. Also, the schema attributes<br/>:db.install/attribute and :db.alter/attribute trigger side<br/>effects, and are never considered redundant.<br/><br/>The Tempid Data Structure<br/><br/>As an alternative to string tempids, peers (but not clients) can<br/>choose to make a structural tempid, allowing them to explicitly<br/>specify a partition for the entity.<br/><br/>You can make a temporary id by calling the datomic.Peer.tempid<br/>method. There are three partitions<br/>built into Datomic.<br/><br/>Partition<br/><br/>Purpose<br/><br/>:db.part/db<br/><br/>Schema partition, used only for attributes and partition entities<br/><br/>:db.part/tx<br/><br/>Transaction partition, used only for transaction entities<br/><br/>:db.part/user<br/><br/>User partition, for application entities<br/><br/>The :db.part/db partition should only be used for schema entities, like<br/>attributes and partitions, not for application data.<br/><br/>The :db.part/tx partition should only be used for transaction<br/>entities, which are created automatically for each committed<br/>transaction.<br/><br/>You should use :db.part/user for your application's entities, or you should<br/>create one or more partitions of your own, as described in Schemas.<br/><br/>This code gets a new temporary id in the :db.part/user partition.<br/><br/>temp_id = Peer.tempid(":db.part/user");<br/><br/>Each call to tempid produces a unique temporary id.<br/><br/>There is an overloaded version of tempid that takes a negative number as a<br/>second argument. This version of tempid creates a temporary id based<br/>on the number you pass as input. If you invoke it multiple times with<br/>the same partition and negative number, each invocation will return<br/>the same temporary id. This can be useful when constructing<br/>transactions that add references between entities, as explained below.<br/><br/>In some cases, you may want to store the literal representation of a<br/>transaction in a file. Acest document descrie tranzacțiile în detaliu.<br/><br/>Structura tranzacțiilor<br/><br/>Datomic reprezintă cererile de tranzacție ca structuri de date. Aceasta este o diferență semnificativă față de baza de date SQL, unde cererile sunt trimise ca șiruri. Folosind datele în loc de șiruri de caractere, este mai ușor să construiți cereri programatic.<br/><br/>O tranzacție este pur și simplu o listă de liste și/sau hărți, fiecare dintre ele fiind o declarație în tranzacție.<br/><br/>Formulare de listă<br/><br/>Fiecare listă pe care o conține o tranzacție reprezintă fie adăugarea sau retragerea unui fapt specific despre o entitate, atribut și valoare, fie invocarea unei funcții de date, așa cum se arată mai jos.<br/><br/>Formulare hartă<br/><br/>Fiecare hartă conține o tranzacție echivalentă cu un set de una sau mai multe<br/>Operațiunile :db/add. Harta poate include o cheie :db/id care identifică datele entității la care se referă harta. Poate include, de asemenea, orice număr de perechi de atribute, valori.<br/><br/>{:db/id entity-id value value atribut valoare atribut<br/>... }<br/><br/>Intern, structura hărții se transformă în structura listei.<br/>Fiecare pereche atribut/valorizare devine o listă :db/add. Dacă nu ați furnizat un id entitate, Datomic va genera un id entitate temporar pentru dvs.<br/><br/>Structura hărții este acceptată ca o comoditate atunci când se adaugă date. Ca o comoditate suplimentară, cheile de atribut din hartă pot fi fie cuvinte cheie, fie șiruri.<br/><br/>Entități de identificare<br/><br/>Există trei moduri de a specifica un id de entitate:<br/><br/>un id temporar pentru o nouă entitate care este adăugată în baza de date<br/><br/>un id existent pentru o entitate care este deja în bază de date<br/><br/>un identificator pentru o entitate care este deja în baza de date<br/><br/>Acestea sunt descrise în secțiunile de mai jos.<br/><br/>ID-uri temporare<br/><br/>Când adăugați date la o entitate nouă, o identificați utilizând un id temporar. ID-urile temporare se rezolvă la ID-urile entității reale atunci când se procesează o tranzacție.<br/><br/>Crearea de tembele<br/><br/>Cel mai simplu tip de id temporar este un șir care urmează aceste reguli:<br/><br/>un id temporar nu poate începe cu un caracter cu două puncte (:)<br/><br/>ID-ul temporar "datomic.tx" identifică întotdeauna tranzacția curentă<br/><br/>Alte stringuri care încep cu "datomic" sunt rezervate pentru utilizare ulterioară de către<br/>Datomic<br/><br/>În datele tranzacției de mai jos, id-ul temporar "jdoe" este folosit pentru a indica faptul că cele două date sunt ambele despre aceeași entitate:<br/><br/>[[:db/add"jdoe":person/first"Jan"]<br/>[:db/add"jdoe":person/last"Doe"]]<br/><br/>Rezoluție ID temporară<br/><br/>Când se procesează o tranzacție care conține ID-uri temporare, fiecare ID-ul temporar unic este asociat cu un ID de entitate reală. Dacă un ID temporar dat este utilizat mai mult de o dată într-o anumită tranzacție, toate instanțele sunt asociate cu același id de entitate reală.<br/><br/>În general, ID-urile temporare unice sunt cartografiate cu ID-urile noii entități.<br/>Cu toate acestea, există o excepție. Când adăugați un fapt despre o entitate nouă cu un id temporar, iar unul dintre atributele pe care le specificați este definit ca :db/unique:db.unique/identity, sistemul va mapea id-ul temporar la o entitate existentă dacă există una cu același atribut și valoare (actualizare) sau va crea o nouă entitate dacă nu există (inserați). Toate adițiile suplimentare în tranzacție care se aplică aceluiași id temporar se aplică entității "upperted".<br/><br/>ID-uri de entitate existente<br/><br/>Pentru a adăuga, modifica sau retrage date despre entități existente într-o tranzacție, trebuie să cunoașteți ID-ul entității. Puteți recupera un id de entitate specific interogând baza de date pentru o cheie externă (consultați Schema pentru informații despre cheile externe).<br/><br/>De exemplu, această interogare recuperează ID-ul unei entități existente pe baza unei adrese de e-mail.<br/><br/>[:find ?e :in $ ?email :where [?e :person/email ?email]]<br/><br/>Dacă ID-ul entității returnat de interogare este 17592186046416, următoarele date de tranzacție vor seta statutul de client al entității:<br/><br/>{:db/id 17592186046416<br/>:client/status:active}<br/><br/>Dacă entitatea în cauză are un identificator unic, puteți specifica id-ul entității utilizând o ref căutare. În loc să interogați baza de date, puteți furniza atributul unic, perechea de valori corespunzătoare entității pentru care doriți să afirmați sau să retrageți un fapt. Rețineți că o ref căutare specificată într-o tranzacție va fi rezolvată de către tranzactor.<br/><br/>{:db/id [:client/email"joe@example.com"]<br/>:client/status:active}<br/><br/>Identificatori de entitate<br/><br/>Sistemul definește un atribut special, :db/ident, care poate fi folosit pentru a atribui un identificator de cuvânt cheie unei entități date. Dacă o entitate are un<br/>Atributul :db/ident, valoarea sa poate fi folosită în locul id-ului entității.<br/><br/>Acest mecanism este ceea ce vă permite să vă referiți la atribute, partiții, tipuri etc., prin specificarea cuvintelor cheie.<br/><br/>De asemenea, puteți utiliza :db/ident pentru a defini entitățile care reprezintă valori enumerate care pot fi apoi menționate prin nume (așa cum este descris în Schemă).<br/><br/>În exemplul de mai jos, entitatea este specificată de către ident<br/>:persoană/nume:<br/><br/>[:db/add:person/name:db/doc"Numele complet al unei persoane"]<br/><br/>Construirea tranzacțiilor<br/><br/>Această secțiune explică cum să construiți tranzacții pentru a adăuga, modifica și retrage fapte. Atunci când o tranzacție adaugă un atribut de tip de referință la o entitate, trebuie să specifice un id entitate ca valoare a atributului. Identificatorul entității specificat pentru valoare poate fi un identificator temporar (dacă entitatea la care se face trimitere este creată de aceeași tranzacție) sau un identificator real al entității (dacă entitatea la care se face trimitere există deja în bază de date).<br/><br/>Acest exemplu arată reprezentarea literală a unei tranzacții care creează două entități noi, persoane numite "Bob" și "Alice". Fiecare entitate are o referință la cealaltă, conectată folosind id-uri temporare, "bobid"<br/>Toate instanțele unui anumit id temporar din cadrul unei tranzacții se vor rezolva într-un singur id de entitate.<br/><br/>Hărțile anexe în tranzacții<br/><br/>Adesea, un grup de entități legate sunt create sau modificate în aceeași tranzacție. Hărțile imbricate vă permit să specificați aceste entități legate împreună într-o singură hartă. Dacă o hartă de entitate conține o hartă imbricată ca valoare pentru un atribut de referință, Datomic va extinde harta imbricată în propria sa entitate. Expansiunea hărții imbricate este guvernată de două reguli:<br/><br/>Dacă harta cuibărit nu include un :db/id, Datomic va atribui un<br/>:db/id automat, folosind aceeași partiție ca și :db/id a entității externe.<br/><br/>Ori referința la harta imbricată trebuie să fie un atribut component, ori harta imbricată trebuie să includă un atribut unic. Această constrângere împiedică crearea accidentală a entităților ușor de orfane care nu au identitate sau relație cu alte entități.<br/><br/>De exemplu, următoarele date utilizează hărți imbricate pentru a specifica două elemente de linie aparținând unei comenzi:<br/><br/>Cele două elemente de linie nu trebuie să specifice un :db/id, și va obține automat id-uri în aceeași partiție ca entitatea de ordin-id.<br/><br/>Deoarece elementele de linie nu au identificatori unici, puteți deduce că<br/>:order/lineItems trebuie să fie un atribut component. (Acest lucru este rezonabil dat domeniul. Elementele de linie nu au existență independentă în afara comenzilor.)<br/><br/>Hărțile cuibărite sunt adesea mult mai convenabile decât expansiunile plate echivalente. Datele de mai jos arată aceeași ordine de informații și elemente de linie ca trei hărți de entitate independente.<br/><br/>Datomic păstrează valorile datelor în timp și vă permite să interogați valoarea bazei de date dintr-un punct în timp. Aceasta înseamnă că este posibil să recuperați datele chiar și după ce au fost retrase, pur și simplu interogând o valoare a bazei de date din trecut.<br/><br/>Funcții de tranzacție încorporate<br/><br/>:db.fn/retractEntitate<br/><br/>Funcția :db.fn/retractEntity ia un id de entitate ca argument.<br/>Retrage toate valorile atributului în cazul în care ID-ul dat este fie entitatea, fie valoarea, retragând efectiv datele entității și orice referințe la entitate, precum și entitățile care sunt componente ale entității date sunt, de asemenea, retrase recursiv.<br/><br/>Următorul exemplu de date de tranzacție retrage două entități, specificând una dintre entitățile prin id, iar cealaltă printr-o căutare ref.<br/><br/>:db.fn/caz<br/><br/>Funcția :db.fn/cas (comparați-și-schimbați) are patru argumente: un id entitate, un atribut, o valoare veche și o valoare nouă. Tranzacționarul se așteaptă la tranzacții și le procesează în serie. Deoarece tranzactorul nu procesează interogări (ele sunt manipulate în colegi), nu este necesară blocarea și prelucrarea este foarte rapidă.<br/><br/>Trimiterea tranzacțiilor<br/><br/>Odată ce o structură de date tranzacție este construită, o trimiteți tranzactorului prin apelarea datomic. Metoda Connection.transact sau transactAsync. Ambele metode returnează un Future<Map>. Metoda sa get returnează o hartă cu informații despre tranzacție dacă tranzacția comite și aruncă o excepție dacă tranzacția avortează.<br/>În Clojure, puteți utiliza, de asemenea, metoda deref sau @ pentru a obține rezultatul unei tranzacții.<br/><br/>Termene de tranzacție<br/><br/>Metoda tranzacției este sincronă, se așteaptă până la finalizarea tranzacției înainte de a reveni. Metoda tranzacțieiAsync este asincronă, se trimite tranzacția la tranzactor și returnează imediat. Un apel pentru a obține pe Viitorul<Map> returnat de la tranzacție va bloca până la finalizarea tranzacției.<br/>Codul aplicației ar trebui să utilizeze suprasarcină pentru a obține care ia o valoare de timeout exprimată ca un lung și un java.util.concurrent.TimeUnit pentru a se asigura că firul apelant nu blochează pe termen nelimitat.<br/><br/>Când o tranzacție este terminată, partenerul nu știe dacă tranzacția a avut succes și va trebui să interogheze o valoare recentă a bazei de date pentru a afla ce s-a întâmplat.<br/><br/>Monitorizarea tranzacțiilor<br/><br/>Peers pot monitoriza toate tranzacțiile procesate de tranzactorul sistemului. Metoda datomic.Connection.txReportQueue oferă o metodă de a obține o coadă de notificări de tranzacții.<br/><br/>Coada furnizează un raport pentru fiecare tranzacție trimisă în timp ce un coleg este conectat la baza de date, chiar și cele trimise de alți coleg. Rapoartele sunt instanțe ale datomic. Tipul TxReport, care oferă acces la valoarea bazei de date înainte și după ce tranzacția a fost aplicată și la setul de fapte adăugate de tranzacție.<br/><br/>Odată ce codul dvs. primește coada de raportare a tranzacției, rapoartele vor fi adăugate la aceasta. Este responsabilitatea codului dvs. de egalitate să goliți coada. Când ați terminat monitorizarea cozii, trebuie să o eliminați prin apelarea datomic. Metoda Connection/removeTxReportQueue, astfel încât să nu continue să consume memorie.<br/><br/>Acest exemplu arată cum să vă conectați la coada de notificare și să recuperați un raport de tranzacție din aceasta.<br/><br/>coada = conn.txReportQueue(); raport = (Map) coada.poll();<br/><br/>Harta raportului conține următoarele chei, care sunt membri statici ai clasei datomic.Connection:<br/><br/>tasta<br/><br/>Utilizare<br/><br/>DB_BEFORE<br/><br/>Valoarea bazei de date înainte de tranzacție<br/><br/>DB_AFTER<br/><br/>Valoarea bazei de date după tranzacție<br/><br/>TX_DATA<br/><br/>Datoms produse de tranzacţie<br/><br/>TEMPID<br/><br/>folosit pentru a rezolva ID-uri temporare<br/><br/>Membrul TX_DATA al unui raport de tranzacție conține setul de date create de o tranzacție (atât aserțiuni, cât și retrageri).<br/>TX_DATA poate fi folosit ca sursă de intrare pentru o interogare. Interogarea de mai jos utilizează datele tranzacției și valoarea bazei de date după ce tranzacția a fost aplicată pentru a afișa fiecare datom al tranzacției.<br/><br/>Interogarea se așteaptă ca valorile DB_AFTER și TX_DATA ale unui raport de tranzacție să fie cele două surse de intrare, în această ordine.<br/><br/>Consultați Interogare pentru mai multe informații despre utilizarea seturilor de tuple ca surse de intrare în interogări.<br/><br/>Tranzacții reificate<br/><br/>Când tranzactorul procesează o tranzacție, creează o entitate de tranzacție pentru a o reprezenta. În mod implicit, această entitate are un atribut,<br/>:db/txInstant, a cărui valoare este momentul în care tranzacția a fost procesată. Given N input bits, use Knuth's circuits to sort them. Then output the middle bit of the sorted list.
