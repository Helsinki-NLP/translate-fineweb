| The problem of keeping track of hardlinks as you traverse a directory tree<br/><br/>March 1, 2022<br/><br/>One of the reactions to my discovery about rsync not handling<br/>hardlinks by default was surprise<br/>that memory was an issue of concern when doing this. After all, you<br/>might ask, how hard is the problem and how much memory are we talking<br/>about? That depends on what sort of program you are and how thorough<br/>a job you want to do.<br/><br/>A program like du has the simplest case. GNU du's normal behavior<br/>is to ignore hardlinks after the first time it sees them, so all it needs to do is keep track<br/>of enough information to identify a file with more than one link<br/>if it sees it again. Normally this is the file's inode number and<br/>device. Modern Unixes use 64-bit inode numbers and generally 64-bit<br/>device numbers, so you're looking at 16 bytes per hardlinked file.<br/>Since you don't want to do a linear search of a potentially very<br/>large array every time you encounter a hardlinked file, you're also<br/>going to need additional space for some sort of random access data<br/>structure. It's probably not worth keeping track of the link count,<br/>especially since it's also a 64-bit integer on at least one popular<br/>platform (Linux); you're probably losing more space than you gain<br/>back by being able to drop entries when you've seen all their links.<br/>However, all of this space usage is relatively trivial by modern<br/>standards.<br/><br/>(You can reduce the memory needs in most situations by having a<br/>two-level data structure, where you give each separate device number<br/>its own inode lookup structure. This reduces the per-file information<br/>down to one 64-bit number from two, since you're storing the device<br/>number only once instead of once per hardlinked file. Usually you'll<br/>only be dealing with a few device numbers so this is a real win. It does<br/>complicate the code a bit. I don't know if GNU du uses this approach.)<br/><br/>A program like rsync or tar has a harder problem, because it<br/>also needs to keep track of the full file name for some instance<br/>of the hardlinked file (normally it will keep track of the first<br/>one). Full file names can be long, especially if you have something<br/>annoying like a deeply nested directory structure, and given the<br/>additional memory usage you now probably do want to also store the<br/>link count so you can delete entries once you no longer need them.<br/><br/>(The good news is that you don't need to look things up by file<br/>name, so you can continue to only have a fast random access data<br/>structure for the inode number plus the device. And you can still<br/>optimize the handling of device numbers, although this is no longer<br/>so much of a memory saving on a percentage basis.)<br/><br/>The worst case for both du style and rsync style programs is<br/>when you are working with a directory tree where all files have<br/>hardlinks that go outside the tree. In this case, both programs wind up<br/>permanently keeping track of all files in the tree in RAM; even if you<br/>track link count, you'll never see all of the links for any file so you<br/>can never drop any entries. Such directory trees can happen in a variety<br/>of situations, including space-saving backups that hardlink unchanged<br/>files between backup trees.<br/><br/>For scale, a large root filesystem on one of our compute servers<br/>has around 2.5 million files with the text of their full file names<br/>amounting to just over 200 Mbytes. On our NFS fileservers, the filesystem with the most files<br/>has around 7 million of them and a total full filename text size<br/>on the order of 665 Mbytes. On modern machines with multiple gigabytes<br/>of RAM that are often mostly unused, programs like du and rsync<br/>should be able to handle even the very unlikely worst case situation<br/>without blowing up.
