| The other day, I found myself explaining different types of encryption, how they work, and what they are used for to someone in my lab implementing a secure system. During this process, I ended up creating a series of fancy diagrams in draw.io - so I thought I'd write it up into a proper demystification blog post.<br/><br/>To start us off here, let's define encryption. Encryption is the process of transforming a given input block of data (of an arbitrary data) using some kind of secret key into a form that is then completely unreadable. Any adversary obtaining a block of encrypted data encrypted with a suitably strong key (and algorithm) is not able to read or understand the data at all - except perhaps infer its original length.<br/><br/>Conversely, decryption is the process of undoing the encryption process with the same (or different, in some cases) key to get back the original data.<br/><br/>For purpose of this blog post, we will assume:<br/><br/>The encryption algorithms in question are perfect with no known weaknesses<br/><br/>Keys used to encrypt and/or decrypt are very strong and can't be cracked<br/><br/>Each of these are fields in their own right that could quite easily take many blog posts to fully explore.<br/><br/>From the perspective of a developer, there are 3 different basic places one needs to aware of. Others certainly exist, but to avoid making this post too long I'll just be covering the following 3:<br/><br/>Device encryption<br/><br/>Transport layer encryption<br/><br/>End-to end encryption<br/><br/>If there's any other encryption scheme you'd like me to cover, please leave a comment below and I'll try my best to explain it in a separate post.<br/><br/>Device encryption<br/><br/>First up is device encryption. Most modern operating systems for phones and PCs alike support device encryption:<br/><br/>Windows<br/><br/>Linux<br/><br/>Android<br/><br/>iOS<br/><br/>Not sure on macOS since I don't own one, but I'd be surprised if it didn't. The purpose of device encryption is that when the device is powered off, all data is stored physically on disk in an encrypted format, making it unreadable should the device be physically stolen - thereby protecting all data stored on it.<br/><br/>This is accomplished in a layered fashion. Let's explain it with a diagram:<br/><br/>Although they may have different names for it, most operating systems back a concept of a "block device". Such a device is capable of storing a given number of bytes of data. Such devices need not be physical disks: they can instead be virtual. For example, zram presents block devices that store data compressed in RAM.<br/><br/>We can make use of this to encrypt hard drives. An encryption layer such as LUKS on Linux presents a virtual block device to the operating system which encrypts all data written to it before saving them back to some physical disk by which it is backed.<br/><br/>On boot, the encryption layer is initialised by the operating system and it asks the user for a password. Upon being given the correct password, the encryption layer is activated, and the operating system can then both request data blocks from the virtual block device (which causes the encryption layer to fetch the encrypted block from disk and then decrypt it before passing it to the requester) and write data blocks back to the virtual block device (whereby the encryption layer will encrypt the new data block before writing it to disk).<br/><br/>Even operating systems such as Windows (e.g. Bitlocker) and iOS which don't expose block devices in the same way as Linux does, the same principles I've explained here apply.<br/><br/>When the device is powered off, the key that was being stored in memory is wiped (it's stored in RAM, and RAM requires power to store data) and the data is secured.<br/><br/>Transport layer encryption<br/><br/>Another place encryption is commonly encountered in when transferring data to and from remote hosts over the Internet. Since the Internet is untrusted, it becomes rather a problem when one wants to transfer personal information such as passwords, bank card numbers, and location information across the Internet, in that such data could be stolen or modified in transit.<br/><br/>To solve this problem, the Transport Layer Security (TLS) protocol was invented. The purpose of TLS is to provide a secure connection between 2 hosts using authenticated encryption that has the following properties:<br/><br/>Eavesdroppers are unable to read data being transmitted<br/><br/>Attackers are unable to successfully modify any data in transit without the destination host knowing about it<br/><br/>The 2 hosts communicating with each other can verify each other's identities 1<br/><br/>Although TLS itself is a protocol that is usually spoken over TCP, because it provides a generic bidirectional pipe through which any binary data can be transmitted and received, it is commonly used to wrap around other protocols to secure them. Suppose Alice and Bob have a messaging app on their phone that sends messages through an intermediary server (perhaps performing store-and-forward functions), but they do not want the server to be able to read their message. The solution here is end-to-end encryption, which prevents the intermediary server from being able to read the message.<br/><br/>Here's a diagram to explain what I mean:<br/><br/>End-to-end encryption is accomplished by using asymmetric cryptography. Asymmetric encryption - unlike symmetric encryption uses 2 keys instead of 1, and these keys also have to possess special properties, so you can't just generate some cryptographically secure random numbers and call it a day 4.<br/><br/>In asymmetric encryption, you have a public key which can only encrypt data, and a private key which can then decrypt the data. An example of this in practice is GPG, which is extensively used e.g. by apt (the package manager on some Linux systems).<br/><br/>In the diagram above, the sender encrypts the message with the public key that belongs to the receiver. They then send the message to the server, who forwards it on to the receiver. The receiver then decrypts the message with the private key (sometimes called a secret key).<br/><br/>In this way, the server is never able to read the content of the message. If the receiver wanted to reply to the sender, the same would happen in reverse. The receiver would need to ask the sender to securely transmit their public key to them, which they could then use to encrypt a message to send back.<br/><br/>In practice, every client involved in an end-to-end encryption system will generate their own keypair that consists of a public and a private key. They will then advertise their public key to everyone, allowing anyone to encrypt a message that only they can decrypt (an example of this: my GPG key can be found here).<br/><br/>It is important to avoid confusing end-to-end encryption with transport layer encryption. Indeed, end-to-end encryption is absolutely no substitute for transport layer encryption, because an application may for example need to authenticate with the intermediary server before being allowed to transmit end-to-end encrypted messages.<br/><br/>Transport layer encryption:<br/><br/>Allows 2 parties to communicate with each other securely<br/><br/>Does not prevent the receiver from reading received data (even if device encryption is employed)<br/><br/>End-to-end encryption:<br/><br/>Requires 3 parties to be involved in order to be effective<br/><br/>Ensures that 2 parties can communicate securely through an intermediary party<br/><br/>Requires that 2 parties wishing to communicate must first securely exchange their public keys and be confident that the public keys they have received actually belong to the other party they wish to communicate with<br/><br/>Can be significantly complicated to implement<br/><br/>Conclusion<br/><br/>In this post, we've looked at 3 types of encryption, how they work, and when they are useful. To summarise:<br/><br/>Device encryption protects data from physical theft<br/><br/>Transport layer encryption protects data in transit between 2 communicating parties talking to each other directly<br/><br/>End-to-end encryption protects the communications of 2 parties who are talking through 1 or more intermediary parties<br/><br/>Each of these are useful in different situations - and most likely are already solved problems. Do not implement any of these yourself. Just let the user enter a secret that only they know, and then we can check if they still know that secret in the future in order to verify that they are the right person attempting to access the account.<br/><br/>This brings a number of problems:<br/><br/>The user has to remember their password (more on this later)<br/><br/>As a service provider, we have to store their password securely so that a hacker can't steal it<br/><br/>Point #1 here is bad enough (more on this later and what you can do about it though), but #2 is a real issue. Because it's repeatable. By making it so that it takes a consistent ~1 second to hash passwords, we can limit our attacker to trying 1 password per second instead of a few million. Furthermore, the bycrypt algorithm doesn't translate onto graphics cards very well at all - further increasing the amount of time it takes to crack our password hashes!<br/><br/>Combined with a salt from earlier, this is a pretty good way of storing passwords.
