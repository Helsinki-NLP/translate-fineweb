Any adversary obtaining a block of encrypted data encrypted with a suitably strong key (and algorithm) is not able to read or understand the data at all - except perhaps infer its original length.<br/><br/>Conversely, decryption is the process of undoing the encryption process with the same (or different, in some cases) key to get back the original data.<br/><br/>For purpose of this blog post, we will assume:<br/><br/>The encryption algorithms in question are perfect with no known weaknesses<br/><br/>Keys used to encrypt and/or decrypt are very strong and can't be cracked<br/><br/>Each of these are fields in their own right that could quite easily take many blog posts to fully explore.<br/><br/>From the perspective of a developer, there are 3 different basic places one needs to aware of. The purpose of device encryption is that when the device is powered off, all data is stored physically on disk in an encrypted format, making it unreadable should the device be physically stolen - thereby protecting all data stored on it.<br/><br/>This is accomplished in a layered fashion. Let's explain it with a diagram:<br/><br/>Although they may have different names for it, most operating systems back a concept of a "block device". Such a device is capable of storing a given number of bytes of data. Such devices need not be physical disks: they can instead be virtual. For example, zram presents block devices that store data compressed in RAM.<br/><br/>We can make use of this to encrypt hard drives. An encryption layer such as LUKS on Linux presents a virtual block device to the operating system which encrypts all data written to it before saving them back to some physical disk by which it is backed.<br/><br/>On boot, the encryption layer is initialised by the operating system and it asks the user for a password. Upon being given the correct password, the encryption layer is activated, and the operating system can then both request data blocks from the virtual block device (which causes the encryption layer to fetch the encrypted block from disk and then decrypt it before passing it to the requester) and write data blocks back to the virtual block device (whereby the encryption layer will encrypt the new data block before writing it to disk).<br/><br/>Even operating systems such as Windows (e.g. Bitlocker) and iOS which don't expose block devices in the same way as Linux does, the same principles I've explained here apply.<br/><br/>When the device is powered off, the key that was being stored in memory is wiped (it's stored in RAM, and RAM requires power to store data) and the data is secured.<br/><br/>Transport layer encryption<br/><br/>Another place encryption is commonly encountered in when transferring data to and from remote hosts over the Internet. Suppose Alice and Bob have a messaging app on their phone that sends messages through an intermediary server (perhaps performing store-and-forward functions), but they do not want the server to be able to read their message. Asymmetric encryption - unlike symmetric encryption uses 2 keys instead of 1, and these keys also have to possess special properties, so you can't just generate some cryptographically secure random numbers and call it a day 4.<br/><br/>In asymmetric encryption, you have a public key which can only encrypt data, and a private key which can then decrypt the data. An example of this in practice is GPG, which is extensively used e.g. by apt (the package manager on some Linux systems).<br/><br/>In the diagram above, the sender encrypts the message with the public key that belongs to the receiver. They then send the message to the server, who forwards it on to the receiver. The receiver then decrypts the message with the private key (sometimes called a secret key).<br/><br/>In this way, the server is never able to read the content of the message. If the receiver wanted to reply to the sender, the same would happen in reverse. The receiver would need to ask the sender to securely transmit their public key to them, which they could then use to encrypt a message to send back.<br/><br/>In practice, every client involved in an end-to-end encryption system will generate their own keypair that consists of a public and a private key. Perhaps you think of some of these rules:<br/><br/>It shouldn't be longer than 32 characters long (no! If longer is an option, take it! )<br/><br/>Change out letters makes it more secure (this can be guessed by attackers)<br/><br/>Here's the big question though: What do any of these password rules achieve? Well, we want to ensure that only the owner of an account can access it.<br/><br/>What could be so bad?<br/><br/>Ok, so we've got our goal. Make sure only the owner of an account can access it. Sounds simple, right? Just let the user enter a secret that only they know, and then we can check if they still know that secret in the future in order to verify that they are the right person attempting to access the account.<br/><br/>This brings a number of problems:<br/><br/>The user has to remember their password (more on this later)<br/><br/>As a service provider, we have to store their password securely so that a hacker can't steal it<br/><br/>Point #1 here is bad enough (more on this later and what you can do about it though), but #2 is a real issue. Because it's repeatable.
