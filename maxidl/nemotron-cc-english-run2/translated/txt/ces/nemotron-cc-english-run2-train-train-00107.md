Both methods return a Future<Map>. Its<br/>get method returns a map with information about transaction if the<br/>transaction commits and throws an exception if the transaction aborts.<br/>In Clojure, you can also use the deref method or @ to get a<br/>transaction's result.<br/><br/>Transaction Timeouts<br/><br/>The transact method is synchronous, it waits until the transaction<br/>completes before returning. The transactAsync method is<br/>asynchronous, it submits the transaction to the transactor and returns<br/>immediately. A call to get on the Future<Map> returned from<br/>transactAsync will block until the transaction completes.<br/>Application code should use the overload for get that takes a<br/>timeout value expressed as a long and a<br/>java.util.concurrent.TimeUnit in order to ensure that the calling<br/>thread never blocks indefinitely.<br/><br/>When a transaction times out, the peer does not know whether the<br/>transaction succeeded, and will need to query a recent value of the<br/>database to discover what happened.<br/><br/>Monitoring transactions<br/><br/>Peers can monitor all transactions being processed by the system's<br/>transactor. The datomic.Connection.txReportQueue method provides a<br/>method to get a queue of transaction notifications.<br/><br/>The queue delivers a report for every transaction submitted while a<br/>peer is connected to the database, even those submitted by other<br/>peers. The reports are instances of the datomic.TxReport type, which<br/>provides access to the value of the database before and after the<br/>transaction was applied, and the set of facts added by the<br/>transaction.<br/><br/>Once your code gets the transaction report queue, reports will be<br/>added to it. It is the responsibility of your peer code to empty the<br/>queue. When you are done monitoring the queue, you must remove it by<br/>calling the datomic.Connection/removeTxReportQueue method so it<br/>doesn't continue to consume memory.<br/><br/>This example shows how to connect to the notification queue, and<br/>retrieve a transaction report from it.<br/><br/>queue = conn.txReportQueue();<br/>report = (Map) queue.poll();<br/><br/>The report map contains the following keys, which are static members<br/>of the datomic.Connection class:<br/><br/>key<br/><br/>usage<br/><br/>DB_BEFORE<br/><br/>database value before the transaction<br/><br/>DB_AFTER<br/><br/>database value after the transaction<br/><br/>TX_DATA<br/><br/>datoms produced by the transaction<br/><br/>TEMPIDS<br/><br/>used to resolve temporary ids<br/><br/>The TX_DATA member of a transaction report contains the set of<br/>datoms created by a transaction (both assertions and retractions). The<br/>TX_DATA can be used as an input source for a query. By default, this entity has one attribute,<br/>:db/txInstant, whose value is the instant that the transaction was<br/>processed. In addition, every datom in a transaction refers to the<br/>transaction entity through the :tx field of datom.<br/><br/>You can add additional attributes to a transaction entity to capture<br/>other useful information, such as the purpose of the transaction, the<br/>application that executed it, the provenance of the data it added, or<br/>the user who caused it to execute, or any other information that might<br/>be useful for auditing purposes.<br/><br/>To annotate the current transaction, simply include an add statement<br/>(either map or list) that uses a temporary id from the :db.part/tx<br/>partition. This capability enables initial imports of<br/>existing data that has its own timestamps.<br/><br/>Redundancy Elimination<br/><br/>A datom is redundant with the current value of the database if there<br/>is a matching datom that differs only by transaction id. If a<br/>transaction would produce redundant datoms, those datoms are filtered<br/>out, and do not appear a second time in either the indexes or the<br/>transaction log.<br/><br/>Transactions are never entirely redundant, because the transaction's<br/>:db/txInstant is always a new fact. Also, the schema attributes<br/>:db.install/attribute and :db.alter/attribute trigger side<br/>effects, and are never considered redundant.<br/><br/>The Tempid Data Structure<br/><br/>As an alternative to string tempids, peers (but not clients) can<br/>choose to make a structural tempid, allowing them to explicitly<br/>specify a partition for the entity.<br/><br/>You can make a temporary id by calling the datomic.Peer.tempid<br/>method.
